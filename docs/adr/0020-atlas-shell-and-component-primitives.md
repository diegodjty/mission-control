# Atlas shell and the component-primitives layer

Status: accepted (refines ADR-0016's Inbox; presentation-only per the UI-redesign PRD)

Mission Control's chrome and visual layer are rebuilt on three decisions, without changing what any view *means*:

1. **The Atlas shell.** The flat top tab bar gives way to the Jarvis-style shell already designed in the owner's claude.ai/design project ("Jarvis-style assistant interface", id `c4327f25-2ba2-49b8-b549-185347143e32`): a persistent slim navigation rail, a header that always names the active **Project**, and a **command palette** (Cmd+K) that jumps to any Project, view, or open issue and invokes the entry-point actions. The shell's decision logic is a pure shared model, **`shell-model`** (view registry, keep-mounted vs remount-on-visit hosting, per-view badge sourcing, active-view transitions), and the chrome itself is an **AppShell** component extracted from the monolithic root — views become siblings the shell hosts. View switching stays local state driven by `shell-model`; **no routing library**. ADR-0004's one-backend/many-**Windows** model is untouched.
2. **A component-primitives layer.** One renderer `components/` library is the ONLY place visual identity lives: headless **Radix** primitives for behavior (Dialog, DropdownMenu, Tabs, Tooltip, Toast — portals, focus traps, keyboard handling, aria wiring) plus hand-rolled Button/Card/Badge/Input in the same pattern, all styled **exclusively** by the existing Atlas design-token custom properties (`index.css`). Dark and light themes render correctly by construction because no primitive owns a color. **No Tailwind, no styled component kit, no new design language**; new dependencies are individual `@radix-ui/react-*` packages only.
3. **One attention surface.** The **Inbox** as a separate destination is folded into a unified cross-project attention surface (a rail entry with a needs-you badge), fed by a single model (**`attention-hub-model`**, absorbing the existing attention and inbox models) so the rail badge, Launcher card badges, and the surface itself always agree. Its semantics — grouping by Project, parked-**HITL**-first urgency, the since-last-seen briefing, clickthrough via the existing switch flow — are preserved; only where it lives and how it looks change. ADR-0016's Inbox intent is *refined, not reversed*. ADR-0012's noise-floor stance carries over unchanged: today's plain tab stays unbadged until the unified surface lands with its single-model badge.

## Why

The five views were reached through a flat tab bar with no hierarchy; cross-project attention was split between Launcher cards and the Inbox tab — two surfaces answering "where am I needed?" differently; every view hand-rolled its buttons, badges, cards, and dialogs, so the app drifted visually; and the layout assumed one desktop window size. Rebuilding chrome on a pure shell model plus one primitives library fixes navigation, consistency, and theming at the root instead of view-by-view — and the primitives make the later per-view rebuilds mostly *deletions* of hand-rolled chrome.

## Considered options

- **Tailwind or a styled kit (shadcn/MUI).** Rejected: imports a second design language beside the Atlas tokens and couples visual identity to a dependency. Radix is behavior-only, so the existing tokens remain the single visual authority.
- **Hand-roll dialog/menu/toast behavior too.** Rejected: focus traps, dismissal, typeahead, and aria wiring are exactly the undifferentiated hard parts Radix tests upstream; our value is the token styling, not re-deriving overlay behavior.
- **A routing library for view switching.** Rejected: views are local state with keep-mounted hosting (live terminals must survive navigation); URL-driven remounting works against the app's core invariant.
- **Keep the Inbox as its own tab and add a rail badge.** Rejected: two surfaces would keep answering the same question with different numbers; one model feeding every badge is the point.

## Consequences

- **The keep-mounted invariant is now pinned by code.** `shell-model` owns "Map, Pane, and Planning survive navigation with live terminals and watchers intact" (a Pane's unmount kills its PTY session); its unit tests and the `shell-keep-mounted` e2e spec fail if the policy drifts. The Launcher and the Inbox stay remount-on-visit deliberately — mounting the Inbox *is* viewing it (it advances the briefing's last-seen stamp).
- **Foundation first, then per-view rebuilds.** The tracer slice (issue 123) shipped the primitives, `shell-model`, the AppShell extraction, and exactly one real dialog — the interrupt guard (issue 114) — replaced end-to-end on the Dialog primitive, proving Radix + tokens + extraction in one pass while the app stays visually and behaviorally unchanged otherwise. The rail, header-with-Project, command palette (`command-registry`), unified attention (`attention-hub-model`), and per-view rebuilds land in follow-up slices against owner-approved mocks; until then the AppShell renders the existing tab-bar visuals.
- **Views may not hand-roll chrome after their rebuild lands.** New modals, menus, toasts, buttons, badges, cards, and inputs come from `components/`; a view needing a new visual extends the primitives library, not itself.
- **The palette adds no new authority.** Every palette command routes through the same flow as its clickable counterpart — project switches through the interrupt guard, actions through their existing entry points — so fast navigation is never less safe than clicking.
- One token addition (`--scrim`, the modal overlay dim) joins the Atlas set so the Dialog primitive stays token-pure in both themes; the palette itself is otherwise unchanged (redesigning it stays out of scope per the PRD).
