## Completed issue 61 — dispatcher-session-block-digest

**What changed** — Mid-drain, the Dispatcher no longer answers "nothing has reported in yet" when Runs have actually finished. The recalibration that quieted the chat had a side effect: the completed-issue reports went only to the on-screen activity log, which the Dispatcher's own conversational session never sees — so it genuinely didn't know what had happened. Now, the moment you ask it something ("where are we at?"), the app slips it a catch-up alongside the status snapshot it already gets: a compact digest of every Run that has reported since it last heard — which issue, whether it completed / got parked for your manual check / hit a blocker, and a one-to-two-line extract of what it said. The chat stays exactly as quiet as before — nothing new arrives unprompted — but the Dispatcher's answers now know what each Run actually did. The digest never repeats runs it already told the session about, and a very long drain is summarized (newest eight listed, the rest acknowledged as a count) so the session's context can't be flooded.

**Try it yourself** — In the QA sandbox (or any project): 1) `cd /Users/devteam/Developer/mission-control && source ~/.nvm/nvm.sh && nvm use 22 && npm run dev` (stop it with Ctrl-C in that terminal). 2) Open the project and start a Drain; let a couple of Runs finish — notice the Dispatcher chat stays quiet while their cards land in the Run log. 3) Type "where are we at?" into the Dispatcher chat. You should see the app inject a bracketed status message into the chat that now ends with a "Completion-block digest" section naming each finished Run and what it reported — and the Dispatcher's reply should name the completed issues (and any Run parked waiting on you) without you telling it anything. 4) Ask again: the same runs are not re-listed; only Runs that finished after your first ask appear in the next digest.

**Verified** — `npm run test` 707/707 (10 new tests: digest inclusion, HITL-park wording, blocked reason, dedupe across asks, unknown-capture exclusion, newest-N cap with elided count, extract truncation, and snapshot+digest composition/ordering/null-model) and `npm run type-check` clean. Not runtime-verified in the live app: the surface is the Dispatcher chat with a live `claude` session mid-drain, which is exactly what walkthrough 58 (HITL, depends on this issue) drives — same convention as sibling issues 59/60.

**Bookkeeping** — Modified `src/shared/dispatcher-status-model.ts` (new pure `buildRunDigest` + `DIGEST_MAX_RUNS` + `RunDigest`; `buildStatusSnapshotMessage` gains an optional digest section, status always first), `src/shared/dispatcher-status-model.test.ts` (+10 tests), `src/renderer/src/App.tsx` (new `dispatcherDigested` ref tracked the same way `dispatcherFed` is and cleared at the same reset points; baselined to the pre-existing Run log when a fresh Dispatcher spawns so only post-seed Runs digest; the on-ask injection composes status + digest and marks ids as given only when the injection actually enqueues). No new chat messages, no ADR-0012 routing changes; blocking notifications (issue 60) untouched. Elided-beyond-the-cap runs are counted as given so a later ask never replays a long drain's history — their full blocks stay in the Run log.

**Doc drift** — `CONTEXT.md` "Dispatcher input contract" says the session's input is a one-time seed "+ a live stream of {Completion blocks read from Receipts, …}". After ADR-0012 + this issue, Completion blocks reach the *session* on-ask (digest riding the status injection), not as a live typed stream — the live stream now feeds only the ambient log/Run-log cards. Amending CONTEXT.md/ADR-0012 is the user's call; walkthrough 58's script should expect the on-ask behavior.
