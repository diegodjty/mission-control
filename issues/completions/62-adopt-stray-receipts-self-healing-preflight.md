## Completed issue 62 — adopt-stray-receipts-self-healing-preflight

1. **What changed** — Mission Control no longer lets one misplaced Receipt file break the whole drain. When a Worker accidentally writes its Receipt into the main checkout (the exact failure that piled up four finished-but-unmerged Runs in the last walkthrough), Mission Control now quietly repairs it: before any merge — and when a solo Run finishes — any stray file in the Receipts folder is automatically committed under its own "chore: adopt stray Receipt(s)" message, and the merge then proceeds normally, with a quiet note in the Dispatcher's activity log saying what was adopted. Anything ELSE left uncommitted (a random edited doc, a stray source file) still stops the merge with the same honest "uncommitted changes on main: <paths>" message as before — unknown state still halts; only the known artifact is repaired. And to attack the cause rather than just the symptom, every Worker's launch prompt now spells out the exact absolute path its Receipt must be written to (its own worktree's copy in parallel mode), so a confused Worker can't misplace the write in the first place.

2. **Try it yourself** — the merge-preflight repair is reproducible in the QA sandbox with plain git, no app needed:
   1. `cd /Users/devteam/Developer/mc-qa-sandbox/repo-a` (any clean git repo with the sandbox reset pattern applied works).
   2. Simulate a misplaced Receipt on main: `mkdir -p issues/completions && echo "stray" > issues/completions/99-stray-test.md` — `git status` shows it untracked (this used to fail every merge).
   3. Start Mission Control (`cd /Users/devteam/Developer/mission-control && source ~/.nvm/nvm.sh && nvm use 22 && npm run dev`), open repo-a, run a parallel drain (2+ concurrent Runs) or any flow that reaches a Merge of finished branches.
   4. What you should see: the merge goes through instead of halting; `git log --oneline` in repo-a shows a `chore: adopt stray Receipt(s) — issues/completions/99-stray-test.md` commit just before the merge commits; the Dispatcher's ambient log shows one "Adopted stray Receipt(s) on main: …" note.
   5. Counter-check the halt still works: `echo x >> docs/PRD.md` (or any non-Receipt file), retry the merge — it refuses with "uncommitted changes on main: docs/PRD.md" and does NOT commit that file.
   6. Also glance at any newly launched Run's Pane: the first prompt line now ends with "Your Receipt path for this Run is exactly /…/issues/completions/NN-slug.md … write your Receipt to that path and nowhere else."

3. **Verified** — exercised end-to-end against real git and the real `afk-merge.sh` in throwaway repos (the integration tests drive the actual binary + script, not mocks): stray untracked Receipt → one adoption commit → clean merge (the exact walkthrough wall); stray *modified* Receipt adopted the same way; dirty non-Receipt path halts with the issue-59 message naming the file and is not committed; mixed dirt adopts the Receipt but still halts on the unknown path; solo finished Run adopts a stray Receipt in its own chore commit while the Run's own work + own Receipt stay in the run commit (issue 59's one-commit contract intact). The Worker-prompt hardening is verified by unit tests on the exact spawned prompt string (solo cwd and worktree cwd variants). NOT runtime-verified: the Electron UI's passive-note line and a live LLM Worker honoring the absolute path — both are exactly what the parked batch QA walkthrough (issue 58) drives; the note wiring is type-checked and uses the existing tested logNote/authority path.

4. **Bookkeeping** — new pure module `src/shared/receipt-adoption.ts` (+ tests): the adopt/halt split (`issues/completions/` only — deliberately not generalized) and the adoption commit message. `src/main/git-worktree-adapter.ts`: new `adoptStrayReceipts` (porcelain with `--untracked-files=all`, pathspec'd commit so foreign staged work is never swept in, skipped mid-merge) and `commitFinishedMain` now adopts strays (excluding the Run's own Receipt) before the run commit. `src/main/run-merge.ts`: `mergeRuns` adopts before the script's preflight; `dirtyMainPaths` now names files inside new directories. `src/shared/ipc-contract.ts`: `adopted` on `MergeRunsResult`/`MainCommitResult`. `src/shared/dispatcher-authority.ts` + `dispatcher-proposal.ts`: new passive `receipt-adopt` action. `src/renderer/src/App.tsx`: passive adoption notes on both the auto-merge and solo-commit paths. `src/main/resolve-run-command.ts` + `pty-session-manager.ts`: the Run prompt carries the absolute per-Run Receipt path derived from the Run's resolved cwd. Tests: 18 added (9 pure-decision, 4 merge-path integration, 2 solo-path integration, 3 prompt); full suite 725 passed, type-check clean. No deviations from the acceptance criteria.

5. **Doc drift** — none. One nuance worth knowing: the issue's "one passive note records the adoption" lands in the Dispatcher's ambient log only on Dispatcher-driven paths (auto-merge / solo drain commit); a purely manual Map-button merge has no Dispatcher log to write to — the dedicated `chore:` commit is the durable record there. This matches ADR-0012's channel model rather than contradicting it.
