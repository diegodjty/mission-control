## Completed issue 72 — runs-on-the-workbench

**What changed** — Mission Control now actually runs work "on the Workbench." For a project whose backlog lives in `~/Workbench/<project>/`, each issue's agent session starts inside the code repo that issue declares (its `repo:` line, or the project's default repo), and the launch prompt tells the agent exactly where the workbench issue files, config, and Receipt destination are — so an agent sitting in a code repo with no `issues/` folder never has to guess. Receipts are watched in one place only (the project's workbench `completions/` folder) instead of per-checkout and per-worktree, and the old "adopt a misplaced Receipt" repair is skipped for these projects since Receipts can't legitimately land in a code repo anymore. Two issues running at once in *different* repos no longer get needless git worktrees — worktree isolation now counts concurrency per repo, and only same-repo concurrency isolates as before. Finally, Mission Control quietly commits the Workbench repo after every Run event — a claim, a park, a done-with-Receipt — with boring messages like `mission-control: issue 05 claim`, never pushing, and never committing workflow artifacts into code repos. Projects opened the old way (a repo with its own `issues/`) behave exactly as before.
