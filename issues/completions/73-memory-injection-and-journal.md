## Completed issue 73 — memory-injection-and-journal

**What changed** — Mission Control now closes the memory loop for Workbench projects, in both directions. Going in: whenever it launches a Worker on an issue or seeds the Dispatcher for a drain, it reads that project's curated memory file (`memory/CORE.md` in the Workbench) and includes it in the agent's opening prompt under a clearly-labeled "Project memory" section — so every agent starts already knowing the project's standing facts. If the memory file is missing or empty, nothing is injected; if it's oversized, it's cut at the ~1.5k-token cap with a visible truncation marker, so a runaway memory file can never flood a prompt. Going out: when a drain ends — for any reason, including you stopping it — the app writes one dated summary into the project's `memory/journal/`: every Run that reported in with its outcome (completed / parked for you / blocked), any doc-drift findings, and notable events like adopted stray Receipts or a Run that finished without a Receipt. A second drain the same day gets its own file (never overwritten), and each journal entry is auto-committed to the Workbench repo with a boring message, same as the other Run-event commits. Old-style (non-Workbench) projects are completely unaffected — no memory dir, both halves inert.

**Try it yourself**
1. `cd /Users/devteam/Developer/mission-control && source ~/.nvm/nvm.sh && nvm use 22 && npm run dev`
2. Open a Workbench-registered project (one whose entry in `~/Workbench/registry.md` is `status: active`), and put a distinctive line in `~/Workbench/<project>/memory/CORE.md` (e.g. "- The demo password is hunter2").
3. Start a Run on any eligible issue → in the Pane, ask the Worker "what do you know about this project from your prompt?" — it should recite your distinctive line (it arrives under the label "Project memory (from this project's Workbench memory/CORE.md …)").
4. Start a drain and let it finish (or click Stop) → within ~5 seconds, look in `~/Workbench/<project>/memory/journal/` — you should see a `YYYY-MM-DD.md` file naming each Run and its outcome, and `git log` in `~/Workbench` shows a `<project>: drain journal YYYY-MM-DD.md` commit. Drain again the same day → a second file `YYYY-MM-DD-2.md` appears; the first is untouched.
5. Open a legacy project (e.g. the QA sandbox at /Users/devteam/Developer/mc-qa-sandbox/repo-a) and repeat → prompts and drains behave exactly as before, no memory anywhere.
